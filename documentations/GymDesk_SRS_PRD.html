1. Introduction

Purpose, scope, and key definitions for the GymDesk platform.

1.1 Purpose

GymDesk is a web-based platform for managing gym memberships, attendance, payments, class schedules, and staff operations — purpose-built for small to mid-sized fitness facilities (1–500 members). This document defines the complete functional and non-functional requirements for the MVP (Minimum Viable Product) and subsequent releases.

1.2 Scope

✅ In Scope (MVP)

  • Authentication & RBAC with 6 roles
  • Member CRUD operations
  • Subscription plan management & lifecycle tracking
  • Attendance via Member ID & QR code
  • Payment recording, history & overdue tracking
  • Trainer profiles & fitness class scheduling
  • Dashboard with key operational metrics
  • Responsive web UI (mobile-first)
  • Bilingual support (Arabic / English)

❌ Out of Scope (Post-MVP)

  • Native mobile apps (iOS/Android)
  • Third-party payment gateways
  • Hardware integrations (turnstiles, biometrics)
  • Marketing automation / email campaigns
  • Online class booking by members
  • Waitlist management
  • Inventory management

2. Product Overview

Vision, target audience, and high-level system architecture.

2.1 Product Vision

Empower every small gym owner to run their facility like a big-box chain — without the complexity or cost.

2.2 Target Audience

  • Primary: Independent gym owners with 1–3 locations, 50–500 members
  • Secondary: Boutique fitness studios, CrossFit boxes, yoga studios, martial arts dojos
  • Tertiary: Multi-location chains (Phase 2)

2.3 High-Level Architecture

The system follows a three-tier architecture with clear separation of concerns:

  • Client Layer: Responsive web application accessible via browser on desktop, tablet, and mobile
  • Application Layer: REST API with modular services (Auth, Member, Subscription, Attendance, Billing, Trainer, Class, Dashboard, Notification)
  • Data Layer: PostgreSQL (primary DB), Redis (cache/sessions), File Storage (QR codes, member photos)

10. Appendix: Glossary

Key terms and definitions used throughout this document.
TermDefinition
Active MemberA member whose subscription end date is in the future and is not suspended/deactivated.
Expired MemberA member whose subscription end date is in the past (beyond any grace period).
Expiring SoonA member whose subscription expires within 7 days.
Grace PeriodConfigurable days after expiry during which check-in is still permitted (default: 3).
Member IDUnique system-generated identifier (e.g., GDM-2026-0001).
Soft DeleteMarking a record as inactive rather than physically deleting it.
QR CodeMatrix barcode encoding Member ID + cryptographic validation hash.
RBACRole-Based Access Control — restricting access based on user role.
RTLRight-to-Left text direction (used for Arabic).
LTRLeft-to-Right text direction (used for English).
P0 / P1 / P2Priority: P0 = Critical (MVP must-have), P1 = Important, P2 = Nice-to-have.
JWTJSON Web Token — compact, URL-safe representation of claims.
Multi-TenantSingle software instance serving multiple customers with data isolation.
