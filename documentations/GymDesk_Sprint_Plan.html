1. Planning Overview & Assumptions

Scope, timeline, sizing methodology, and key planning assumptions.

1.1 Scope of MVP

This sprint plan covers the Minimum Viable Product (MVP) for GymDesk as defined in the SRS/PRD v1.0. The MVP includes all P0 requirements, most P1 requirements, and defers P2 items to post-launch phases.

📐 Sprint Planning Assumptions

  • Sprint Duration: 2 weeks per sprint (10 working days)
  • Team Size: 5-6 people (see Section 10 for breakdown)
  • Story Point Scale: Modified Fibonacci (1, 2, 3, 5, 8, 13, 21) — 1 SP ≈ 4 ideal hours
  • Team Velocity: Estimated ~25–30 SP per sprint (ramping from ~20 in Sprint 0)
  • Tech Stack: React/Vue.js frontend, RESTful API (Node.js/Python), PostgreSQL, Redis
  • Bilingual Foundation: i18n architecture embedded from Sprint 0
  • Testing Baseline: Unit tests required for all P0 features; ≥ 70% coverage target
  • Definition of Done: Code reviewed, tested, documented, deployed to staging

1.2 Prioritization Mapping

3. Sprint 0: Foundation & Infrastructure

Weeks 1–2: Setup project skeleton, CI/CD, database schema, auth system, and developer tooling.

🔧 Sprint 0 — Foundation Layer

Jun 9 – Jun 22, 2026  |  22 Story Points
Goal: Establish project foundations so all feature sprints run smoothly. No user-facing features yet.
Deliverables: Running dev environment, database schema migration, CI/CD pipeline, JWT auth skeleton, i18n framework, shared component library start, project documentation.

🧱 Technical Tasks (Not User Stories)

T-001 Project scaffolding: monorepo setup (frontend + backend + shared), linting, formatting, commit conventions 3
T-002 Database schema design & migration scripts (PostgreSQL): members, subscriptions, payments, attendance, trainers, classes, users, roles, audit_logs — all with gym_id for multi-tenancy readiness 5
T-003 Authentication skeleton: JWT generation/validation (RS256), bcrypt hashing, login/register API stubs, RBAC middleware 5
T-004 CI/CD pipeline: GitHub Actions (lint, test, build, deploy to staging), Docker Compose for local dev 3
T-005 i18n architecture: JSON locale files (en/ar), RTL/LTR CSS strategy, locale detection & switching service 3
T-006 Shared UI component library start: Button, Input, Modal, Table, Badge, Card — all bilingual-compatible 3

4. Sprint 1: Core Operations

Weeks 3–4: Member management, subscription plans, payment recording, and the foundational data layer.

📋 Sprint 1 — Core Operations (Epics 1, 2 + Epic 8 partial)

Jun 23 – Jul 6, 2026  |  39 Story Points
Goal: Build the core data management layer — members, subscriptions, and payments must be fully CRUD-operational.
Deliverables: Member CRUD UI, subscription plan management, payment recording, QR code generation, staff account management.

📌 User Stories

US-001 Create Member Profile — As Reception, I want to create a new member profile so I can register them in the system. (Name, phone, email, DOB, photo, emergency contact, notes, subscription plan, start date) 5
US-002 Search Members — As Reception, I want to search for a member by name, phone, ID, or plan so I can quickly find their profile. (Fuzzy search ≤ 1s) 3
US-003 Edit Member Profile — As Reception, I want to update a member's contact info or membership plan. 3
US-004 Deactivate/Reactivate Member — As Reception, I want to deactivate (soft delete) a member to prevent non-paying access. Owner can hard delete. 2
US-005 Generate QR Codes — As Owner, I want to generate scannable QR codes for new members (Member ID + validation hash) so they can check in quickly. (Printable PDF card format) 5
US-006 Create Subscription Plans — As Owner, I want to create/update/deactivate plans (Monthly/Quarterly/Annual) with name, duration, price, and description. 5
US-007 Record Membership Payment — As Reception, I want to record a payment (amount, method, date, notes) and automatically update the subscription end date. 8
US-008 Renew Membership — As Reception, I want to renew a membership so access continues without interruption. (Extends from current end_date, not renewal date) 5
US-009 View Overdue Payments — As Manager, I want to see all overdue payments with a "Red" indicator (>30 days) so I can follow up. 3

5. Sprint 2: Attendance & Classes

Weeks 5–6: QR check-in engine, attendance tracking, trainer profiles, and class scheduling.

💪 Sprint 2 — Attendance & Classes (Epics 3, 4 + remaining Epic 8)

Jul 7 – Jul 20, 2026  |  33 Story Points
Goal: Members can check in (QR + manual); trainers can manage classes; attendance is tracked end-to-end.
Deliverables: QR scanner UI, check-in flow, attendance history, trainer profiles, class scheduling calendar, class attendance marking.
US-010 QR Code Check-In — As a Member, I want to check in by scanning my QR code so I don't wait at the front desk. (Web-based scanner, ≤ 2s processing, hash validation, forgery rejection) 13
US-011 Manual Check-In — As Reception, I want to check in a member manually (by searching name or entering Member ID) so I can help members who forgot their QR code. 3
US-012 Today's Check-In Count — As Manager, I want to see today's check-in count on my dashboard so I know how busy the gym is. (Near-real-time, ≤ 30s delay) 3
US-013 View Attendance History — As a Member, I want to view my attendance history (filtered by date range) so I can track my gym visits. 3
US-014 Create Trainer Profiles — As Manager, I want to create trainer profiles (name, specialization, phone, email, photo, bio). 3
US-015 Create & Schedule Fitness Classes — As Manager, I want to create class types and schedule recurring sessions on a weekly calendar (day, time, trainer, room, capacity). (Double-booking prevention) 8
US-016 View Weekly Schedule — As a Trainer, I want to view my weekly class schedule so I know what sessions I'm teaching. 3
US-017 Mark Class Attendance — As a Trainer, I want to mark attendance (Present/Absent/Late) for each scheduled class session. 5
US-025 Manage Staff Accounts — As Owner, I want to create, edit, and deactivate staff accounts (Reception, Trainer, Manager roles) so I control system access. 5

6. Sprint 3: Dashboard, i18n & Notifications

Weeks 7–8: Operational dashboards, bilingual support, notification engine, and core reporting.

📊 Sprint 3 — Dashboard, i18n & Notifications (Epics 5, 6, 7)

Jul 21 – Aug 3, 2026  |  34 Story Points
Goal: Management visibility, bilingual usability, and automated member communication.
Deliverables: Dashboard with KPI cards, revenue & membership reports, in-app notifications, complete Arabic UI (RTL), language switching, audit log.
US-018 Dashboard KPIs — As Owner, I want to see Total Active Members, Expired Subscriptions, Today's Check-ins, and Monthly Revenue at a glance. (4 KPI cards + membership expiry timeline for next 7 days) 8
US-019 Revenue Reports — As Manager, I want to view monthly revenue with previous-month comparison so I can understand financial trends. (Bar chart) 5
US-020 Export Reports — As Owner, I want to export reports as CSV/PDF so I can share them externally. 5
US-021 Membership Expiry Reminders — As a Member, I want to receive an in-app notification when my subscription is about to expire (7 days & 1 day before) so I can renew on time. 5
US-022 VIP Expiry Alerts — As Manager, I want to be notified when a high-value member's subscription is expiring so I can personally reach out to retain them. 3
US-023 Arabic UI (RTL) — As an Arabic-speaking Owner, I want the full system UI in Arabic with RTL layout so my staff and I can use it comfortably. (All screens translated, RTL tables, reversed column order) 8
US-024 Language Switching — As bilingual Reception staff, I want to switch between Arabic and English instantly (no page reload) so I can assist members in their preferred language. 5

7. Sprint 4: Security, Polish & Release

Weeks 9–10: Security hardening, edge cases, performance optimization, bug fixes, and production readiness.

🔒 Sprint 4 — Security, Polish & Release

Aug 4 – Aug 17, 2026  |  18 Story Points
Goal: Harden the system for production — security audit, edge cases, performance testing, and bug fixes. Prepare for public launch.
Deliverables: Penetration test results, rate limiting, audit log UI, all P1 edge cases handled, staging → production deployment, launch documentation.
US-026 Audit Log — As Owner, I want to see an immutable audit log of all staff actions (login attempts, payment changes, profile edits) so I can monitor for suspicious activity. 5
US-027 Password Security — As System Admin, I want passwords hashed with bcrypt (cost factor ≥ 12) so member data remains secure. 2
T-007 Security Hardening — Rate limiting on login (5 failed / 15 min), HTTPS enforcement, input sanitization (XSS/SQLi), JWT rotation, RBAC server-side audit 5
T-008 Edge Case Implementation — Code all P1 edge cases: duplicate check-in window, grace period logic, trainer double-booking, expired member rejection, DST/timezone handling, queue for rush-hour check-ins 3
T-009 Performance & Load Testing — Load test with 50 concurrent users, verify P95 response times, optimize DB queries, verify 3G page load < 3s 3

  • ✅ Audit logging for all critical operations
  • ✅ Security: rate limiting, input sanitization, HTTPS enforcement

📋 Recommended Ceremonies

  • Sprint Planning: First Monday of each sprint (2 hours) — commit to sprint backlog
  • Daily Standup: 15 minutes (async via Slack/Teams is acceptable)
  • Sprint Review / Demo: Last Thursday of each sprint (1 hour) — showcase to stakeholders
  • Sprint Retrospective: Last Friday (1 hour) — inspect & adapt
  • Backlog Refinement: Mid-sprint (1 hour) — story point estimation, break down large stories
